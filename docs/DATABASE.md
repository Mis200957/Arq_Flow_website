## RLS Policies Summary

### businesses
- `SELECT`: owner (`owner_id = auth.uid()`) OR admin
- `INSERT`: public (via service role in API route)
- `UPDATE`: owner can update own fields; admin can update all

### payments
- `SELECT`: business owner OR admin
- `INSERT`: public (service role)
- `UPDATE`: admin only

### profiles
- `SELECT`: own profile OR admin sees all
- `UPDATE`: own profile only

### messages / conversations / orders / customers / escalations
- `SELECT`: business owner (`businesses.owner_id = auth.uid()`) OR admin
- `INSERT`: service role (via n8n or API routes)
- `UPDATE`: service role

### notifications
- `SELECT`: own notifications (`user_id = auth.uid()`)
- `UPDATE`: own notifications (marking read)

### automation_logs / audit_logs
- `SELECT`: admin only
- `INSERT`: service role

### app_settings
- `SELECT`: admin only
- `UPDATE`: admin only

## Helper Functions

### `is_admin()`
```sql
CREATE OR REPLACE FUNCTION public.is_admin()
RETURNS boolean AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.profiles
    WHERE id = auth.uid() AND role = 'admin'
  );
$$ LANGUAGE sql SECURITY DEFINER;
```

### `owns_business(business_uuid uuid)`
```sql
CREATE OR REPLACE FUNCTION public.owns_business(business_uuid uuid)
RETURNS boolean AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.businesses
    WHERE id = business_uuid AND owner_id = auth.uid()
  );
$$ LANGUAGE sql SECURITY DEFINER;
```

### `increment_usage(p_business_id, p_messages_delta, p_tokens_in_delta, p_tokens_out_delta, p_cost_egp_delta)`

Atomically increments monthly usage counters. Creates a new period row if none exists for the current month.

```sql
CREATE OR REPLACE FUNCTION public.increment_usage(
  p_business_id    uuid,
  p_messages_delta int     DEFAULT 1,
  p_tokens_in_delta bigint DEFAULT 0,
  p_tokens_out_delta bigint DEFAULT 0,
  p_cost_egp_delta numeric DEFAULT 0
) RETURNS void AS $$
DECLARE
  v_period date := date_trunc('month', now())::date;
BEGIN
  INSERT INTO public.usage_counters
    (business_id, period_start, messages_sent, tokens_in, tokens_out, cost_egp_total, last_message_at)
  VALUES
    (p_business_id, v_period, p_messages_delta, p_tokens_in_delta, p_tokens_out_delta, p_cost_egp_delta, now())
  ON CONFLICT (business_id, period_start) DO UPDATE
    SET messages_sent   = usage_counters.messages_sent   + EXCLUDED.messages_sent,
        tokens_in       = usage_counters.tokens_in       + EXCLUDED.tokens_in,
        tokens_out      = usage_counters.tokens_out      + EXCLUDED.tokens_out,
        cost_egp_total  = usage_counters.cost_egp_total  + EXCLUDED.cost_egp_total,
        last_message_at = now();
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

## Multi-Tenancy Pattern

ArqFlow uses a **shared database, tenant-filtered** pattern:

1. Every tenant table has a `business_id` column
2. RLS policies enforce `business_id IN (SELECT id FROM businesses WHERE owner_id = auth.uid())`
3. n8n workflows use the service role key (bypasses RLS) and always include `business_id` in all writes
4. The `TENANT_CONFIG` node injected into every cloned bot workflow carries the `business_id` as a constant — no Airtable/Sheets lookup at runtime
5. Admins see all data via `is_admin()` check in RLS policies

This means:
- A bug in one tenant's bot cannot expose another tenant's data to end-users
- Clients logging into the dashboard only see their own conversations, orders, and analytics
- Admins can query across all tenants without any extra configuration
